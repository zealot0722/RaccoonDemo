# Raccoon AI Support Demo 最終交付回答

## 第二題

### 釐清問題

1. AI 客服主要服務的客群是既有客戶、新客戶，還是內部客服人員？
2. 客戶會從哪些入口進來，例如網站、LINE、Email、App 或社群私訊？
3. 客服需要支援哪些問題類型，例如 FAQ、查訂單、查貨態、退換貨、保固或客訴？
4. 哪些情境必須轉真人客服，例如客訴、退款、資料不完整、低信心回答或高價商品？
5. 商品推薦需要依照哪些條件判斷，例如預算、用途、品類、庫存、價格區間或過往購買紀錄？
6. 商品資料來源在哪裡，是否有正式商品 DB、圖片、價格、庫存與商品詳情連結？
7. FAQ 來源是固定文件、資料庫、知識庫，還是需要從既有客服紀錄整理？
8. 是否需要查訂單或貨態？若需要，系統要接哪個訂單、物流或 ERP API？
9. 真人客服接手後要如何通知客服人員，例如後台、Email、LINE Notify、Slack 或 CRM？
10. AI 回答是否需要留下判斷紀錄，例如 intent、confidence、命中的 FAQ、推薦商品與轉人工原因？
11. 是否需要讓使用者在對話結束後評分？評分資料是否會用於後續優化？
12. 是否有個資、金流、醫療、法務或退款等高風險內容需要特別限制？
13. 上線初期的流量預估是多少？是否需要限制使用量、登入或試用碼？
14. MVP 要先做可展示 demo、內部試用，還是要直接接正式客服流程？
15. 成功指標是什麼，例如 FAQ 自助解決率、轉人工率、推薦點擊率、CSAT 或客服節省時間？

### User Flow / Story

客戶進入客服頁後，先看到友善的招呼訊息，直接輸入問題或點選預設問題。系統會將訊息送到後端，由 LLM 做第一層語意分類，再由本機 guardrail 修正明顯錯誤分類，例如價格條件、查貨態、退貨、亂碼或多意圖。若是 FAQ，系統查詢 FAQ 資料並回覆流程；若是商品推薦，系統依 DB 商品價格、用途與庫存推薦 1 到 3 個商品；若是查貨態，系統要求訂單或物流單號，資料足夠時回覆貨態；若是退貨或客訴，系統會要求必要資料或直接建立需人工處理工單。

客服人員進入後台後，可以看到工單列表、狀態分類、緊急程度、對話紀錄與 AI 判斷原因。客服可以更新工單狀態、調整分級，並新增 mock 客服回覆。客戶表示沒有其他問題後，系統才顯示評分流程，將 CSAT 寫回資料庫。

### Feature List

- 客戶聊天頁：支援 FAQ、商品推薦、退貨、查貨態、客訴、轉人工、閒聊與亂碼處理。
- Groq 語意分類：辨識 `faq`、`product_recommendation`、`return_request`、`order_status`、`complaint`、`human_handoff`、`unclear`、`chitchat`、`conversation_end`。
- 本機 guardrail：修正價格上下限、多意圖、純符號、亂碼、貨態與退貨關鍵字。
- FAQ 查詢：由 `faq_articles` 依關鍵字命中並回覆固定流程。
- 商品推薦：由 `products` 查詢，依預算、用途、品類與排除條件推薦符合 DB 價格的商品。
- 商品詳情頁：提供圖片、中文名稱、原文名稱、價格、庫存、適合情境與商品連結。
- 查貨態：由 `order_statuses` 查詢 demo 貨態資料，之後可替換成正式物流 API。
- 退貨流程：缺資料時要求送貨貨號、姓名、電話號碼，可附照片；資料齊全後轉人工。
- 工單紀錄：將 tickets、messages、ai_decisions、csat_feedback 寫入 Supabase。
- 客服後台：提供工單分類、狀態更新、緊急分級、AI 判斷與 mock 客服回覆。
- 評分流程：使用者明確表示沒有其他問題後才顯示評分，評分寫回 DB。
- 部署保護：使用 `DEMO_ACCESS_CODE` 避免公開 URL 被無限制使用。

### Edge Cases

- 使用者只輸入「推薦商品」但沒有預算或用途：不直接推薦，先追問。
- 使用者輸入「1000 以上」：只推薦價格大於等於 1000 的商品。
- 使用者輸入「1000 元內」：只推薦價格小於等於 1000 的商品。
- 使用者輸入「1000 到 2000」：同時符合上下限才推薦。
- 使用者追問「有其他的嗎」：根據上一輪 AI 推薦內容排除已推薦商品。
- 使用者輸入 `?`、`??`、純符號或 `123123123`：分類為 unclear，不顯示商品卡。
- 連續三次無法辨識需求：提示若無待處理問題，本次對話將關閉。
- 使用者同時說「我要退貨，順便查貨態」：先查可處理的貨態，再補問退貨資料。
- 貨態查詢缺訂單或物流單號：要求提供訂單編號或物流單號。
- 退貨缺資料：要求送貨貨號、姓名、電話號碼。
- 明確真人、人工、客訴：客戶端只顯示「請稍後，客服人員將很快為您服務。」
- DB 沒有符合商品：不讓模型硬編商品，回覆請調整預算、用途或品類。
- 使用者點商品詳情後返回客服頁：對話保留，不重置。

### 設計邏輯

#### Prompt 設計

本 demo 將 LLM 拆成兩個角色：分類與回覆。第一步由 Groq 判斷使用者 intent、confidence、tone、missing fields、keywords、budget 與是否需要人工。第二步不是完全相信模型，而是在後端加入 deterministic guardrail，修正可由規則明確判斷的情境，例如價格範圍、查貨態、退貨、亂碼、純符號、多意圖與上下文追問。

回覆生成採「資料先行」原則：FAQ 必須來自 `faq_articles`，商品必須來自 `products`，貨態必須來自 `order_statuses`。模型不能自行創造商品、價格或政策。對客戶的文字統一使用「您」，並移除 markdown `**` 等不適合客服聊天的格式。

#### 判斷邏輯 Criteria

- `faq`：政策或流程問題，例如退貨規則、付款方式、配送時間、保固。
- `product_recommendation`：使用者想找商品、比較商品或提供預算用途。
- `return_request`：使用者實際要申請退貨或換貨。
- `order_status`：使用者要查貨態、查件、問貨在哪。
- `human_handoff`：使用者明確要求真人、人工或客服人員。
- `complaint`：使用者表達客訴、不滿、服務很差或要求申訴。
- `unclear`：亂碼、純符號、無意義數字或無法判斷的內容。
- `chitchat`：打招呼或無明確客服需求的閒聊。
- `conversation_end`：使用者明確表示沒有其他問題，才進入評分。
- `multi_intent`：同一句包含多個需求時，先處理可處理部分，再補問缺資料；不一律轉人工。

#### 範例 Input / Output

Input: `請問商品可以退貨嗎？`  
Output: `您可以參考以下流程：先確認商品狀態與訂單資料，再提供送貨貨號、姓名、電話號碼...`

Input: `我想找 1000 元內的新手入門商品`  
Output: `依照您的需求，為您推薦以下選項。P001｜入門保養組...價格：NT$ 890...詳情連結：/products/P001`

Input: `1000以上的`  
Output: 僅推薦價格大於等於 1000 的商品，例如 `P002｜行動辦公耳機`。

Input: `我的包裹到哪了`  
Output: `可以，我幫您查貨態。請問您方便提供訂單編號或物流單號嗎？`

Input: `RAC1004的東西在哪`  
Output: 回覆訂單 `RAC1004` 的目前貨態、位置、預計到貨時間與補充說明。

Input: `我要退貨`  
Output: `請提供您的送貨貨號、姓名、電話號碼...收到必要資料後，我會把退貨申請轉交客服人員確認。`

Input: `我要找真人客服`  
Output: `請稍後，客服人員將很快為您服務。`

Input: `?` 或 `123123123`  
Output: `請您重新敘述您的問題，或補充需要協助的事項...`

#### 待改進內容

- Groq Free tier 有 rate limit，正式展示或多人試用應改用付費 API key 或加上排隊與快取。
- 客戶使用頁面與客服人員使用頁面目前為了展示方便放在同一個 demo 專案中，正式版應拆成客戶端與客服端兩套入口，並補上總控管、權限、帳號與營運監控功能。
- 商品資料目前是 demo catalog，之後應接正式商品、庫存與價格 API。
- 查貨態目前使用 demo `order_statuses`，之後應串正式物流或 ERP。
- 客服回覆目前只寫入 demo DB，尚未串 LINE、Email、CRM 或通知工具。
- 圖片上傳目前作為附件紀錄，尚未做 AI 圖像辨識。
- 後台目前是 mock console，正式版需要登入權限、角色控管、操作紀錄與審計。
- CSAT 已有 DB schema，之後可做回覆品質分析與 prompt 調整報表。

### 選用該 AI 工具來 Vibe Coding 的原因

本次使用 Codex 協助 Vibe coding，原因是它適合在既有 repo 中快速讀取檔案、修改程式、補測試、執行 smoke test 並推送到 GitHub。相較只產生片段程式碼，Codex 可以直接維護整個專案結構，包含 Vercel API、前端 UI、Supabase schema、測試腳本與文件。

採用 Codex 也與模型能力和使用成本有關。最新 GPT-5.5 模型在實作、推理與修正能力的評估上，不比 Claude Opus 4.6 差；但在可用量與使用成本上，Codex / GPT 系列對這類長時間開發任務更有彈性。若是實際開發場景，理想做法會是依任務分派不同模型，例如用強推理模型處理架構與 bug，用較快模型處理重複性修改與文件整理。

模型與平台選型如下：

- Groq：目前主要基於成本與部署速度考量，適合作為 MVP 的語意分類與回覆生成服務。後續可依穩定性、成本與資料控管需求，改成或並行使用本地部署的 Ollama、Gamma，或使用 GPT、Gemini 作為更可靠的語意分析工具。
- Vercel：提供公開 URL、Serverless API Routes 與 GitHub 自動部署，不需要本機 n8n 長時間運行。
- Supabase：提供 Postgres DB，能保存 FAQ、商品、工單、訊息、AI decision 與 CSAT。
- Codex：負責快速完成端到端 demo、修正 bug、建立測試、整理文件與部署流程。

整體設計重點是客戶的易用性：讓客戶盡量用自然語言或一般口語表達方式完成自己的需求，例如查詢政策、推薦商品、查貨態、申請退貨或轉接真人客服。內部 AI 判斷與工單處理則放在客服後台，不把 intent、confidence 等內部資訊暴露給客戶。
